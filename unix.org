* Unix Overview

* The Unix Programming Environment
** overview

*** Terminal and shell interaction                           :terminal:shell:
(from llama4)

1. Terminal emulator creates a new pseudo-terminal (PTY) pair.
2. Forks a new process, which becomes the shell (e.g., =bash=, =zsh=).
3. The child process (shell) is connected to the PTY slave device.
4. The terminal emulator process (parent) is connected to the PTY
   master device.


Here's a simple representation:

- Terminal Emulator (parent)
	+ Forks → Shell (child)
		- Connected to PTY slave
	+ Remains connected to PTY master, displaying output and
          receiving input

The shell then runs in the terminal, reading input from the user,
executing commands, and displaying output.

*** I/O model
- Most commands take input from a file or read the standard input.
- Output is sent to the standard output
- Error messages are sent to the standard error
- Pipes are used to connect programs  
- Input can be redirected from a file or received through a pipe
- Output can be redirected to a file or sent to a pipe

*** commands
date, who, mail, stty, ed, ls, cat, pr, mv, rm, wc, sort, tail, cmp,
diff, pwd, grep, cd, rmdir, echo, nohup, at, kill

** The File System                                         :files:filesystem:
*** Detecting EOF                                        :setuid:set:uid:eof:
  - There's no special character to represent end of a file
  - The =read= kernel subroutine used to read from files returns the
    number of bytes read
  - When the bytes read is zero, end-of-file is assumed
  - ctrl-d sends all currently pending input to the program. If
    there's no pending input, the program assumes end of file
    e.g. logging out of the shell using ctrl-d

- Unix defaults to text file formats with no assumptions except
  newline termination
  - Makes programs interoperable
  - Avoids the need for specialized set of programs for each file format
- The current working directory is a property of each running process
  in the system
- Each logged in user is assigned a user id and group id
  - Relevant files: =/etc/passwd= and =/etc/group=
- The =set-uid= bit e.g. on the =passwd= command allows any user
  executing a program run it as the program's owner (allowing users to
  change own passwords).
- A directory is a file that is written to using special system routines
- Directory permissions are different from file permissions:
  - read permission allow listing directory contents
  - write permission allow adding/removing directory content
  - execute permission allows searching directory content

*** Inodes                                                     :inode:inodes:

- An inode is an entry to a table that stores file metadata such as
  name, time created or modified, length of the file, location of the
  file, etc
- =ls -i= show a files inode number
- =ln= creates a different link to the file (different name for same
  file).  A file is deleted when it has zero links. (inode is shared)
- =mv= changes the existing file link (name) (inode remains the same)
- =cp= creates a new file with identical content as the original file
  (new inode and link created)

*** Devices

- Peripheral devices are represented as files in the /dev subdirectory.
  - Instead of special routines to access these devices, programs read
    from and write to them as files
  - In the kernel, references to the device files are translated to
    hardware commands
  - =ls -l= shows the type of device (character or block) as the first
    character of the mode
  - The mount command shows mapping of storage devices to directories
- the =tty= command shows the user's current controlling terminal in /dev
  - =mesg= can be used to alter permissions on the terminal e.g. to
    allow/disallow other users from sending messages via =talk= or
    =write=
  - =/dev/tty= is a synonym for a user's controlling terminal that
    programs can use to send messages
- writing to =/dev/null= discards the data and reading from it gives
  zero bytes.

*** commands
od, du, crypt, ls -c, ls -u, ls -t, ls -i, tty, mesg, write
      
** The Shell                                                          :shell:
*** Command line structure

- =;= terminates commands
- =()= group commands
- =|= creates a pipeline and has a higher precedence than =;=
  - =date; who | wc= has pipeline taking precedence
  - =(date; who) | wc=  has =;= taking precedence

- =&= terminates command without waiting for output, turns it into a
  background job
- =tee= copies input to a named file as well as its output
- the various special characters like =;, >, <, |, &= are interpretted
  by the shell and not sent to programs as arguments. The following
  are equivalent:
  - =echo hello >junk=, =>junk echo hello=.
    - The program =echo= never sees =>=

*** Metacharacters      
- Special characters that are interpreted by the shell before
  executing commands
- Their interpretation can be escaped using quotes or backslashes
- Single and double quotes escape each other
  
*** Creating new commands
- chmod x makes files executable by the shell
- custom programs are set in PATH to be discoverable
- $1 to $9 matches the (upto 9) arguments to a command
- $* matches all the arguments provided
- $0 is the name of the command itself being executed by the subshell
- backtics allow a program's output be used as arguments

*** Shell variables

- Defined for the current shell and not accessible in sub shells
  unless exported
- the =.= operator can be used to read commands from a file to execute
  in the current shell without using a sub shell
- Variables can be temporarily exported to sub shells by declaring
  them inline before the command

*** I/O Redirection                                             :redirection:

- Every program has 3 file descriptors by default when it starts,
  standard input, standard output and standard error, represented by
  number 0, 1 and 2 respectively
- n>filename redirects the output from the file descriptor n to filename
- n>&m merge output from file descriptors n and m
- n<&m merges the input from file descriptor n with m
- the =here document= construct can be used to specify the standard
  input of a program along with the program itself
  #+begin_src bash
  wc -l <<End
  once upon a time,
  there were 3 little bears
  End
  #+end_src

*** Looping                                                             :for:

- use the =for= statement:

#+begin_src bash
for i in *
do
	     echo $i
done
#+end_src

- =do= and =done= appear either at the start of a line or after a semi-colon:

#+begin_src bash
for i in *; do echo $i; done
#+end_src

output of the for loop can be redirected/piped.

** Filters
*** grep family                                                        :grep:
- for each input line matches a pattern defined as a regular
  expression and prints it as output
- grep :: matches ed's regular expressions
- egrep :: supports more general expressions like or with | and
  parentheses
- fgrep :: supports parallel searching of patterns but without
  metacharacter support
- =egrep= and =fgrep= support -f flag to search patterns specified in
  a file in parallel
*** Other filters
- sort, comm, tr, uniq, dd
*** sed                                                                 :sed:
- A streaming version of ed (stream editor) that supports ed's commands
- A more general grep like tool that performs an action (print, delete, write etc)
  on lines that match certain patterns
- Sample commands:
 `sed 'commands' filenames` - Apply commands to files
 `sed -f cmdfile filenames` - Read commands from file
 `sed -n` - Suppress automatic printing (use with `p`)
 `sed '20,30p'` - Print lines 20-30 (with `-n`)
 `sed '1,10d'` - Delete lines 1-10
 `sed '/pattern/d'` - Delete lines containing pattern
 `sed '/pattern/!d'` - Delete lines NOT containing pattern
 `sed '/^$/,/^end/p'` - Print from empty line to line starting with "end"
 `sed '$d'` - Delete last line
 `s/old/new/` - Substitute first occurrence in line
 `s/old/new/g` - Substitute all occurrences in line
 `s/old/new/p` - Print line if substitution made
 `s/old/new/w file` - Write changed lines to file
- Limitations of sed:
  - limited memory, makes only one pass, no forward references, no
    manipulation of numbers (pure text processor)
*** awk                                                                 :awk:
- fixes some of the limitations of =sed=
- has the same form =awk 'program' filenames..=
- program can be a sequence of instructions in the following form
  sequencially performed on each line:
  
  =pattern { action }=
  =pattern { action }=

- Either of pattern and action is optional. When no pattern is
  specified, it matches all lines. When no action is specified,
  the default is to print matched lines
- By default, lines are broken into fields divided by whitespace.
  - Each is assigned to variables $1 to $NF. NF is the number of fields
  - A single character separator can alternatively be specified with option =-F=
- =print= and =printf= can be used for printing.
  -  =print= simply outputs comma separated fields separated by the
    default whitespace separator
- BEGIN actions are perfomed before processing the first line.
- END actions are performed after processing all the lines
- Richer filter functionality includes arithmetic, conditionals, control flow,
  arrays, associative arrays

** Shell Programming                                                  :shell:
- A shell program can be used to provide a more intuitive interface
  over existing commands
- A case statement can be preferred over an if statement if the =test=
  command is not a shell builtin because it would be faster.
- in addition to =for= loops, there exists =while= loops and =until= loops
- Piping the whole loop to a command will prevent multiple invocations
  of the command per loop
- You can loop forever by using =:= as the while loop condition.
- using curly brackets around a shell variable can allow extra processing e.g.
  - ${var-thing} # value of var if exists, otherwise thing
  - {var+thing}  # thing if var is defined, otherwise nothing
  - {var?message} # if defined, $var, otherwise print message and exit shell
- =trap sequence-of-commands list-of-signal-numbers= can be used to handle
  signals like hangup, interrupt, terminate (except kill -9)
- To exit after handling a signal with =trap=, an explicit =exit=
  command is added to its sequence of commands
- =exec= replaces the current process with the named program avoiding
  the need to start a sub-process.
- =shift= advances the program arguments by one to the left, so that
  $2 becomes $1 and so on.
- =$*= and =$@= expand into arguments passed to command. Blanks in a single
  argument results in multiple arguments
  - ="$*"= is a single word composed of all arguments
  - ="$@"= expands to arguments passed to command. Blanks in a single
    argument are retained.
- The shell demonstrates iterative programming based on re-use of
  existing programs that cooperate well with each other because of
  shared conventions on how information is shared and communicated.
** Programming with Standard I/O                                      :stdio:
- When we've reached the limits of what can be done with the shell, we can consider writing
  programs in C.
  - Usually, a first version in shell helps define an interface
- header files are stored in =/usr/include=
- When a file is opened in a C program, it is given an internal name,
  the file pointer, which points to a structure that contains
  information such as location of a buffer, the current character
  position in that buffer and the operation, read or right.
- when a program is started, 3 file pointers defined in <stdio.h> are
  already defined, the stdin, stderr and stdout. These are constants
  that cannot be reassigned
- A file is usually buffered so that it is written in large chunks for efficiency.
  Calling =fclose= forces it to be written to disk
- It is often better to write programs that to a basic thing well than
  add features to existing programs.
  - Easier to maintain, less buggier and easier to memorize usage.
** Unix System Calls                                               :syscalls:
*** Low level IO
- Low level I/O is mostly through reading and writing files including
  with peripherals
- A FILE pointer points to a structure that contains among other
  things a file descriptor to the file.
  - the macro =fileno= in stdio.h returns the descriptor
- When a program is started by the shell, it inherits 3 file
  descriptors by default
  - 0 for stdin, 1 for stdout and 2 for stderr
  - Normally they are attached to the terminal unless redirected by
    the shell
- Input and output is done via two system calls =read(2)= and
  =write(2)=
  - each take a file descriptor as first argument, an array of bytes
    (source or dest) as the second argument and number of bytes to be
    transfered as the 3rd
- Reading and writing in chunks that match the disk blocksize may be
  more efficient than character at a time.
  - block size is defined as parameter BUFSIZ in stdio.h
- Syscall =open(2)= is used to open files for reading and writing. It
  returns a file descriptor.
- Syscall =creat(2)= create a file if it does not exist. If it exists,
  it will be truncated
  - It accepts permissions that will be used to create the file if it
    doesn't exist
- syscall =unlink(2)= removes a file from the filesystem
- When syscalls encounter errors, they set the variable =errno= in
  errno.h
  - A string representation of the error can be found using
    =strerror(3)= in string.h
- Normally reading and writing happens sequentially in files. However,
  the syscall =lseek(2)= can be used to randomly access a file
*** inodes
- A directory is a file containing a list of filenames and anindication of
  where they are located.
  - A file entry consists of an inode number (index into the inode
    table) and a filename
    - found in =/usr/include/dirent.h= in linux
  - managed by functions like =opendir(3)=, =readdir(3)=, =closedir(3)= found in =dirent.h=
- The =stat(2)= and =fstat(2)= syscalls are used get information about a
  file from the inode table
  - =stat= returns inode info from a filename
  - =fstat= returns inode information from a file descriptor
  - the information is saved in a structure =struct stat= from <sys/stat.h>
- Many programs are not system programs but still use information
  maintained by the OS. Such information exists in standard header
  files which the programs include, instead of defining it themselves,
  to help with portability.
*** Processes  
- The =system(3)= command from =<stdlib.h>= executes a command in a subshell
  - It forks then the sub-process execs itself with the shell command
- The =exec(3)= comes in two variants:
  - =execlp= for arguments known in advance
  - =execvp= for arguments now known in advance
- =exec(3)= overlays the process it is running in with the command to execute
  until it exits, unless the call to exec fails
- To regain control of the original process, we need to split the
  process into two copies and wait for the child to finish executing.
- =fork(2)= is used to split processes. In the child, the call to
  fork returns 0, while in the parent , it returns the process_id of
  the child
  - Typically, an if block where the return is zero is used to execute the child process
- =wait(2)= is used by the parent process to wait for state changes in child processes.
  - whenever any child process exits, it will return its process id.
- A child process inherits the file descriptors of its parent, i.e. they both are connected
  to the same files.
  - Typically, the parent would flush stdout before forking so that
    it's output comes out before the child's
- The syscall =dup(2)= duplicates a file descriptor on the lowest
  unallocated file descriptor number.
  - It can be used to reallocate file descriptors by calling =close=
    afterwards on the initial file descriptor
  - This can be used e.g. for have a child process access the terminal (/dev/tty)
*** Signals and Interrupts
- A program needs to deal gracefully with:
  - program faults e.g. illegal memory references, floating point
    errors, peculiar instructions
  - outside world signals e.g. interrupt, quit, hangup (loss of
    connection), terminate (from kill command)
- When such an event occurs, a signal is sent to all processes started
  from the same terminal
- The default is to terminate the process, unless a signal handler is
  set
- the system call =signal(2)= alters the default action
  - the first argument is the signal to handle
  - the second argument is the address of a function to handle the
    signal, or a code to specify whether the signal is ignored or is
    given the default action (termination)
  - The return value is the is the previous signal handler for the
    signal
- For programs that are started non-interactively (started with &),
  the shell arranges for them to ignore signals by default
- A program like a text editor may interpret a signal as a request to
  stop what it is doing and return to its own command processing loop.
  - This program may utilize =setjump(3)= to save the state of its
    stack just before the loop, and a signal handler that calls
    =longjump(3)= to goto the saved state
- Programs that are waiting for I/O may handle interrupts by setting
  an error via =errno=
  - when execution resumes where it was interrupted, the error can be
    handled as desired.
- Since signals are sent to all processes in the terminal, some that
  start subprocesses programs may opt to ignore interrupts until their
  subproceses return
- The system call =alarm(2)= allows one to set the signal =SIGALRM= to
  be sent to the process after a given number of seconds, e.g. to
  ensure some work is done within a given amount of time
** Programs Development
- =yacc= is a parser generator, converts a BNF grammar into parser that
  will parse statements in the language. Steps:
  - grammer is specified
  - each rule can be augmented with an action in C. This defines the
    semantics
  - A lexical analyzer reads input and provides it to the parse as
    tokens. This is implemented as a user defined function called =yylex=
  - yacc creates the parser as a function named =yyparse=
  - yacc then produces a C file that can be compiled normally
  - On encoutering syntax errors, =yyparse= will call user defined
    =yyerror=
